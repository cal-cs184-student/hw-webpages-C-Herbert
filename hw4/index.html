Part 2: Simulation via numerical integration

Comparison of parameters

The first constant I tried adjusting was ks. I found that this parameter controlled the stiffness and elasticity of the cloth. Low values resulted in a cloth that bounced and sagged more, while high values produced a cloth that maintained its shape better. Also, at values around 200,000 N/m, the system started to demonstrate noticable instability, and began 'waving' on its own.

Next up was density, which controls how much mass each point represents. This constant affects the system in an inverse manner to the ks constant. Increasing the density parameter makes the cloth look less stiff for a fixed ks value. Conversely, decreasing the density resulted in a cloth that looks more stiff, as it maintains its shape better while falling. At around 0.3 g/cm^2, and all other parameters at their default values, the waving phenomenon came back.

Finally, the damping constant. High values made the cloth fall more smoothly, bending more gradually and presenting fewer 'fine' wrinkles as it came to a rest. I found it especially interesting how larger folds slowly revealed themselves from the cloth, without the 'noise' caused by smaller bounces and stretches. Low damping values made the cloth swing and bounce more rapidly, typically overshooting its rest state and oscillating about the pinned axis for longer. Ripples on the cloth surface are stronger and last for longer with a low damping value, which provides enough time for them to add together and produce a chaotic, noisy pattern over the surface.

Part 3: Handling collisions with other objects

My implementation for sphere collision is fairly straightforward. First, I check if the point mass is inside the sphere by comparing the sphere's radius with the magnitude of the difference between the point mass's position and the sphere's origin. I should note that, in my implementation, I opted to compare the square of the radius and the square of the magnitude of the difference vector, since this avoids a sqrt operation and produces equivalent results in our case. If the point mass is inside the sphere, I find the tangent point it should've intersected at by normalizing the difference vector from before, multiplying the result by the sphere's radius, and finally adding the sphere's origin. This yields the desired tangent point in world-space coordinates. Next, a correction vector is calculated as the difference between the tangent point and the point mass's last_position. Per its computation, this is the vector that must be added to the point mass's last_position to get the tangent point. If we instead add a fraction of this value, the resultant position will approach and drag along the surface, rather than snapping to it, which produces a sort of friction. Thus, to add a friction effect, we multiply the correction vector by \(1-f\) before adding it to the last_position and assigning the result to position.

My plane collision implementation is slightly more complicated. To determine if the point mass has intersected the plane, I compare the signs of two dot products. The first is the dot product of the difference vector between the point mass's last position and the plane's origin point with the plane's normal vector. The second is the dot product of the plane's normal and the difference vector between the point mass's current position and the plane's origin. We'll call this difference vector diff_now. If the sign changes between these two dot products, we know one difference vector makes an angle of less than 90 degrees with the plane's normal, while the other makes an angle greater than 90 degrees. Thus, the point must have crossed the plane during the last time step.

Once we've determined a point has crossed the plane, the tangent point can be calculated by subtracting diff_now's projection onto the plane's normal vector from the point mass's position. Before proceeding with the correction vector calculation, I add a small offset to the tangent point, towrads the side of last_position, with a magnitude of SURFACE_OFFSET. Then, the reamining correction vector, friction, and position calculations proceed as in the sphere case.

Part 4: Handling self-collisions

To handle self-collisions, my implementation uses a hash map based on the locations of the point masses. When properly implemented, this accelerates collision detection algorithm by quickly ruling out many mutually distant pairs of point masses. However, such a map requires a hash function, which was the first piece I implemented. My hash function works by first establishing a virtual grid of buckets, each having dimensions \(w = 3 * \frac{width}{num_width_points}\), \(h = 3 * \frac{height}{num_height_points}\), and \(t=max(w,h)\). For a given 3D position, the hash function places it in the appropriate bucket by rounding each axes to the nearest multiple of the corresponding bucket dimension. This is done by subtracting the fmod of an axis' position from the position itself, using the relevant bucket dimension as the divisor. Then, from the rounded coordinates, we calculate an arbitrary hash value. For my function, I went with x + y * z, since it seems to provide a better distribution than simply adding or subtracting the axes. Some constants could be inserted to make the value more unique, but that would require more operations, and I wanted a performant hash function since it will be called many times.

From here, the rest of the implementation is fairly easy. At each time step, we reconstruct the hash map in the build_spatial_map function by iterating over the point masses. For each point mass, we calculate the hash value using the previously defined function and check whether the map has an existing list for that hash value. If so, we simply insert the new point mass. Otherwise, we construct a new list, insert the point mass, then add the list to the map.

Finally, to actually check for self-collisions, we begin by querying the hash map for all point masses sharing a hash value. For each of these other masses, we check if the distance between it and the current mass is less than or equal to \(2* thickness\) using the difference vector between them. Again, I applied the squaring trick for closeness checking from the sphere collision section here. If the condition is satisfied, we add a correction vector that moves the current mass's position to be exactly \(2* thickness\) away from other to an accumulation vector. After dividing by the simulation step factor, the accumulation vector is then applied to the current point mass. Its also worth noting that a check needs to be performed to prevent a mass from performing a check with itself. I did this by just comparing the addresses of the point masses.
